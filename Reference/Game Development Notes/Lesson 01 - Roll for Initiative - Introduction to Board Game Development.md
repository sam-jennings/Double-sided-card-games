# Lesson 01: Introduction to Board Game Development

## TL;DR
- Game development is an 11-step pipeline from concept through sales; focus on one step at a time rather than the whole
- First prototype should be playable in days/weeks, not beautiful; push past perfectionism to reach the table
- Iterative playtesting cycle (prototype → test → iterate) is where games genuinely come together
- Don't start art until mechanics are solid; form factor resolves organically through playtesting
- Minimum development cycle is ~1 year; budget accordingly with dedicated focus
- Follow your passion over trends; be a trendsetter not a trend follower
- IP licensing without publisher backing is extremely difficult; start with original concepts

## Core Concepts

- **Vision Document** — Captures aspirational moments: specific moments players will remember and celebrate at the table. Establishes pillars (foundational elements that must survive) before mechanics are designed. Example: Battlestar Galactica brainstorm focused on "traitors among crew, not knowing who to trust."

- **Playable Prototype** — Minimum viable version that tests core mechanics. Doesn't need artwork, polish, or even complete rules; needs only to validate that core ideas work conceptually. Solo-playtest first before external playtests.

- **Iterative Playtesting** — Cycle of test → observe → update → test again. Many designers take detailed notes during playtests (filling notebook pages) then apply changes to next version. Shift from observation + participation (early) to pure observation (later).

- **Form Factor** — Board size, component count, box dimensions. Resolves organically through playtesting rather than planned upfront. Include in mock budget early so cost implications are understood.

- **Development Roles** — Designer (original concept, prototype, initial testing); Developer (further mechanical refinement, additional content, testing); Producer (coordinates all roles, manages timelines); Graphic Designer (UI: cards, boards, tokens, box); Artist/Illustrator (visual art, cover); Editor (clarity, structure); Proofreader (typos, grammar); Playtester (external feedback); Production (manufacturing); Sales/Marketing. One person often fills multiple roles in small projects.

## Design Principles

- If core mechanics don't work after iteration → shelf the prototype, return later with fresh eyes
- If starting with perfectionism → delay actual playtesting, which is essential for real progress
- Push past first-playtest failure → expected and normal; make up rules on the fly rather than getting stuck
- Don't start graphic design/art → until you're confident mechanics won't change significantly
- Avoid chasing IP licensing → without established publisher backing; typically adds approval friction
- Decide on observation-only vs. participating playtests → early: participate to understand experience; later: observe only
- Form factor questions → don't answer prematurely; let playtesting reveal whether box feels too big/small
- Failed prototypes → shelf them as normal practice; successful designers have shelves of "failed" versions

## Heuristics (Fast Decision Rules)

- Timeline gut-check: If designing assumes <1 year, recalibrate expectations
- Scope warning: If multiple prototype iterations don't converge core vision with playable experience, consider setting aside
- Art readiness: If you're still making major mechanic changes, don't start art yet
- First prototype deadline: Days to one week, using blank cards / existing components
- Randomness choice: Different players prefer different randomness (dice vs. cards); test feeling with your target audience rather than debating theoretically

## Design Frameworks

### 11-Step Development Pipeline
1. **Concept (Vision Document)** — Aspirational moments, core pillars, emotional goals
2. **First Prototype** — Playable in days/weeks; test conceptually, not aesthetically
3. **First Playtest** — With others; identify broken core mechanics; expect failure
4. **Iterate (Steps 4-5)** — Update → Test → Update cycle; many cycles needed
5. **Graphic Design & Art** — Only once mechanics stable; start ~10 months before printing
6. **Editing & Proofreading** — Structural clarity, then typos/grammar
7. **Printing** — Factory, print-on-demand, or self-publish depending on scale
8. **Marketing** — Can start before printing; intensifies after physical product exists
9. **Selling** — Fulfillment, distribution, revenue collection

### Trend vs. Original Design Decision Tree
- If design follows trend (deck-building, nature theme, legacy): high speed needed to capitalize before market saturates; high competition makes differentiation hard
- If design follows passion (you genuinely love the concept): sustainable motivation, more likely you'll finish despite setbacks

## Common Pitfalls

- **Starting beautiful too early** → Artwork becomes sunk cost; significant mechanic changes force art redo/scrap. Fix: Keep prototype minimal until mechanics are proven stable.

- **Waiting for perfect prototype before external playtesting** → Perfectionism delays actual feedback. Players surprise you; solo testing only goes so far. Fix: Push "done enough" prototype to table quickly.

- **Assuming form factor must be decided upfront** → Costs and dimensions reveal themselves through playtesting (does game take too much table space? too cramped?). Fix: Start with rough estimate, refine as feedback arrives.

- **Designing for trends instead of passion** → Trend market saturates; you're competing against faster-moving designers. If not passionate, unlikely to sustain through year+ development. Fix: Design game you'd want to play.

- **Licensing IP without publisher** → Major IP holders inaccessible to solo designers; adds approval steps and complicates pitch. Fix: Build original concept first; offer to adapt to IP later if publisher interested.

- **Assuming randomness is wrong** → Some games thrive with dice, others with controlled card hands. Different player preferences. Fix: Test how randomness feels in your specific game; don't dogma-choose.

## Application

**When starting a new concept:**
1. Write vision document: What table moments do you want? What's the emotional core? What are your pillars?
2. Prototype in days (blank cards, sketched board, borrowed dice)
3. Solo-playtest (ensure parts move, basic flow works)
4. External playtest with 1-2 players (expect something breaks)
5. Iterate based on feedback (not perfection, just informed changes)
6. Playtest cycle 4-10+ times until core feels solid
7. Only then start art/graphic design
8. Budget 1 year minimum for entire process

**Deciding when to start graphic design:** Ask "Are core mechanics still changing?" If yes, wait. If stable for 2+ playtests, start design work.

**Managing multiple roles:** For solo/small designer: Designer + Developer + Producer at minimum. Hire/collaborate for graphic design + art. Can do proofreading yourself if detail-oriented.

**Handling failed prototypes:** Common and healthy. Shelf immediately; don't force. Return in 6+ months with fresh perspective.

- Battlestar Galactica case study: Top-down design starting from "traitors among crew" theme; hidden information mechanic (face-down cards) emerged from asking "how do we hide actions?"
- First prototype expectations: Should take days/week, not weeks/months; solo test first; expect external playtest to reveal broken mechanics
- Role flexibility: Most early projects have one person filling 3-4 roles; specialization comes later
- Trend analysis: Trends emerge from genuinely novel experiences (Risk Legacy legacy mechanic, Wingspan nature theme); following trends requires speed and differentiation; following passion is more sustainable
- Software choices: InDesign is industry standard but expensive; Affinity is viable free alternative; Tabletop Simulator useful but not replacement for physical playtesting

- Randomness preferences: Player-dependent (some hate dice, others fine with shuffling); should serve game's theme and emotional goals; test with target audience
