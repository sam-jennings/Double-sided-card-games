# Lesson 10: Crafting Clear Game Instructions

## TL;DR
- Start writing rulebook after first successful playtest as an outline; writing reveals design problems early.
- Update every 2 weeks, not after every playtest. By external testing, must be fully playable (rough Word doc acceptable).
- Structure: WHO (player context) → WHAT (goal) → HOW (mechanics) → Competitive/Co-op/Teams?
- Game flow (round/turn structure) before specifics. Establish broad context before painting in details.
- Two rulebook purposes (learning vs. reference) are in tension. Solutions: FAQ/clarification section, separate Learn-to-Play + Rules Reference books, or robust alphabetical index.
- Clarity rules: avoid unneeded words, use simple language, break up text walls, explain why. Rulebooks remain essential—videos won't replace written rules.
- Three criteria for success: Does first page convey experience? Can players quickly find info during play without frustration? Are players more excited after reading?

## Core Concepts
- **Rulebook Timing** — Start outline early (after first playtest) to reveal design problems. A rule taking a page to explain one card = card is too complex. Update regularly (2-week intervals), not after every playtest.
- **Rulebook Necessity** — Yes, even with videos. Not everyone learns visually. Rulebooks are marketing tools. That one person teaching everyone else will read the rulebook.
- **Game Flow Foundation** — Establish round structure and turn order BEFORE explaining specifics. Example: "Multiple rounds, each player takes turn clockwise, check win condition, next round" sets context for all details.
- **Point of View** — Second person ("you") = casual, welcoming, but hard with asymmetric roles. Third person ("the player") = precise, works with roles, but wordy. Avoid gendered pronouns either way.
- **Two-Purpose Tension** — Rulebook optimized for learning walks step-by-step. Rulebook optimized for lookup includes all edge cases alphabetically. Can't optimize for both simultaneously.
- **Clarity over Complexity** — Simple games like Ticket to Ride are gold standard partly because game is simple. Complex games demand equal clarity within complexity. Length doesn't excuse poor structure.

## Design Principles
- If writing rules → reveal overly complicated mechanics and simplify design before external testing
- If explaining mechanics → establish game flow framework first, then paint in details
- If choosing point of view → use second person for casual games, third person for asymmetric/role games
- If players struggle to understand → problem is usually rulebook structure, not reader comprehension
- Avoid unnecessary words ruthlessly → brevity aids understanding and memory retention
- Break walls of text with visuals → fatigue sets in with half-page paragraphs; chunk information
- Explain why mechanics exist → "Action cards represent tactical choices" beats just listing steps
- Two rulebooks → teach first game, reference during play. Must not contradict; teach players to use Reference during play

## Heuristics (Fast Decision Rules)
- **The One-Page Rule**: If explaining one card takes a full page, the card/mechanic is too complex. Redesign.
- **Learning vs. Reference**: If you can't answer "is this learn-to-play or reference rulebook?", design both separately.
- **Text Wall Test**: If paragraph takes >5 lines, break it. Use subheaders, sidebars, examples, visuals.
- **Clarity Check**: Read first page. Do you understand the experience, win condition, and game type without reading further? If no, rewrite opening.
- **Font Rule**: 10-11pt font. Translators need room to expand text; don't go below 9pt.
- **The Sidebar Trap**: Don't bury critical rules in sidebars. Reference them explicitly in main text: "See sidebar page 12."

## Design Frameworks
- **Rulebook Structure (Canonical order)**:
  1. Title & context (WHO are players? WHAT is their goal? HOW do they accomplish it? Type: competitive/co-op/teams?)
  2. Component list (pictures + quantities, defer functionality explanation)
  3. Setup (numbered steps with large visuals, simple and concise)
  4. Game flow (round/turn structure before specifics)
  5. Turn actions & mechanics (now that flow is clear, explain what players do)
  6. End game (win condition, final scoring if applicable)
  7. Clarifications/FAQ (weird interactions, edge cases from testing)
  8. Index or reference (alphabetical lookup for complex games)

- **Point of View Decision Matrix**:
  - Second person ("you"): Family games, simple games, inclusive tone. Avoid in asymmetric/role games.
  - Third person ("the player"): Asymmetric games, role games, precision needed. Wordy but clear.
  - Avoid gender pronouns unless distinguishing specific players in examples.

- **Two-Purpose Solution Options**:
  1. **FAQ/Clarification Section**: Main rules + clarification section in back (best for medium-weight games).
  2. **Two Rulebooks**: Learn-to-Play (preset boards, simplified) + Rules Reference (alphabetized, complete, lookup-optimized). Teach players which to use when.
  3. **Robust Index**: Single rulebook with detailed alphabetical index (subsections under each letter). Works for complex games if index is truly comprehensive.

- **Clarity Audit (3-step check)**:
  1. Read first page aloud. Does it convey experience clearly? → If not, rewrite opening.
  2. Pick 5 random rules. Can someone look them up in 30 seconds without sighing? → If not, reorganize or add index.
  3. After reading rulebook, are players excited or overwhelmed? → Excitement = success. Overwhelm = structure problem.

- **Golden Rule for Edge Cases**: Include catch-all for situations not covered: "If a situation is not covered, resolve it least beneficially to the player causing it" (Arkham Horror LCG approach).

## Common Pitfalls
- **Writing too early** → Full rulebook before design is finalized. Fails because: design changes make it obsolete, creates work waste. Fix: Wait until first playtest succeeds, then write outline. Update every 2 weeks, not after every session.
- **Explaining component function in list** → Component section describes what each piece does. Fails because: players learn names but don't understand mechanics; repeats explanation later. Fix: Component list shows pictures and quantities only. Defer function explanation to rules.
- **Flow buried in details** → Jumping into action types before explaining round structure. Fails because: reader is lost; doesn't know how turn/round/game end works. Fix: Establish broad structure first (round → turns → actions → end check), then explain specifics.
- **Wall-of-text paragraphs** → Half-page paragraphs of unbroken text. Fails because: readers fatigue, forget what they read. Fix: Break into 2-3 sentence chunks. Use subheaders, bullets, sidebars, visuals.
- **Mixing learning and reference** → One rulebook trying to teach first game and serve as lookup reference. Fails because: two different readers (learner vs. in-play question) have different needs; one format doesn't optimize for both. Fix: Separate into two rulebooks or use robust index.
- **Sidebar traps** → Critical rules buried in sidebars. Fails because: many readers ignore sidebars (trained by ads/banners). Fix: Reference explicitly in main text: "See sidebar on page 12 for command sheet details."
- **Assuming first draft is final** → No revision, no clarity check. Fails because: author knowledge blindness—you know how it works, so it reads clear to you. Fix: Have someone unfamiliar playtest with rulebook, ask for confusion points.

## Application
- **When designing**: Write rulebook outline after first successful playtest. As you write, ask: "Is this mechanic easy to explain? If it takes a paragraph, simplify it."
- **For learning games**: Preset board + "STOP — Advanced rules in back" = teach first game, enable confident play before complexity.
- **Asymmetric games**: Use third person ("the player") or establish point of view in examples ("You are the Vanguard"). Second person breaks when multiple roles exist.
- **Balancing learning vs. reference**: Single rulebook = FAQ section. Multiple rulebooks = separate Learn-to-Play and Rules Reference, teach players which to consult when playing.
- **Writing for clarity**: Avoid jargon, use simple language, break up text, explain why. Test with someone unfamiliar. Reread and trim every unneeded word.
- **Video integration**: QR codes linking to tutorials are supplementary, not replacement. Written rules remain essential for accessibility and sales marketing.
