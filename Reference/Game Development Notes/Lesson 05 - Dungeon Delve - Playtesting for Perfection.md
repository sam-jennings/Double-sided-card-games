# Playtesting for Perfection

## TL;DR
- Build playable core mechanics in 1-2 days to learn what your game actually is through play
- Four testing stages: Solo Dry Run → First Playtest → Local Playtesting → External/Blind Playtesting, each serving different diagnostic purposes
- Expect 10+ significant iterations; don't balance until external phase, when players want to play the game
- Designer participation is valid in local testing (reveals problems firsthand), step back in blind tests for fresh perspective
- Separate clarity problems from design problems: try better explanation first, mechanical change second
- Cheating reveals misunderstandings (accidental) or unintuitive design; phone-checking signals pacing issues
- Late-stage contradictory feedback and solution-oriented feedback can be safely ignored; focus on root problems instead

## Core Concepts

- **Playtesting Stages** — A sequence where each stage has different diagnostic value. Solo Dry Run validates mechanics; First Playtest tests fun + clarity; Local Playtesting gathers extensive iteration data; External/Blind Playtesting provides fresh eyes unclouded by designer intent.

- **Designer Blindness** — After 12 months with a game, designers cannot see what's actually unclear to fresh players. External testing becomes mandatory for valid feedback.

- **Balance vs. Fun** — Balance testing is premature in early stages. Don't balance a game nobody wants to play yet. Move balance focus to external playtesting phase when core mechanics are solid.

- **Feedback Signal vs. Noise** — Phone-checking = pacing issue. Repeated confusion on one mechanic = clarity problem, not design problem. Contradictory feedback = less actionable than consistent feedback.

- **Prototype Iteration** — Playing reveals what mechanics actually do better than theory. Each iteration teaches something new; radical changes are normal and valuable.

## Design Principles

- If players are confused by a mechanic repeatedly → explain it better before changing it mechanically
- If a mechanic serves intentional purpose but feels "wrong" → clarify its purpose in rulebook/cards first
- If players consistently disengage at same point → pacing issue, not fun issue
- Avoid defending design choices when feedback contradicts them; listen without attachment
- Avoid heavy balance focus until external phase; internal phase is about discovery
- Prioritise internal testing with 2 players first, expand to target count only after core validation
- When receiving feedback, always determine clarity problem vs. design problem first

## Heuristics (Fast Decision Rules)

- 1-2 days to prototype core mechanic = ready to learn what you actually made
- Four feedback questions after every playtest: favorite part? least favorite? confused by what? fun? why/why not?
- Explicitly ask for honest negative feedback — people default to politeness
- Cheating + children's unintentional rule misuse = redesign signal
- Win/loss tracking + power tracking in external phase = proper balance data
- Contradictory feedback from different groups → less urgent than consistent feedback
- One playtest card requiring one full rulebook page = redesign the card, not the system

## Design Frameworks

**Four Playtesting Stages Framework**:
1. Solo Dry Run: Verify mechanics work, estimate session length, catch obvious flaws
2. First Playtest: Diagnostic, not judgmental; expect something to break; improvise past problems
3. Local Playtesting: Observe carefully; note frustrations, confusion, disengagement; take detailed notes on player behavior
4. External/Blind Playtesting: Game must stand alone without designer; fresh perspectives reveal unclear rules and balance issues

**Feedback Analysis Framework**:
- Is this a clarity problem (explain better) or design problem (change mechanics)?
- Is this consistent feedback (multiple playtesters) or contradictory (some love it, some hate it)?
- Is this about the core experience or a finished element (late-stage feedback)?
- Is this a root problem or a proposed solution?

**Observation Checklist for Local Testing**:
- Which mechanics created excitement?
- Where did players check phones (pacing/engagement issue)?
- Which cards/roles/factions won most often?
- What confused multiple players at the same point?
- What special ability generated excitement (should be emphasized)?

## Common Pitfalls

- **Balancing too early**: Designers balance games nobody wants to play yet → wasted effort on problems that don't matter if mechanics aren't fun
- **Getting defensive about feedback**: Closing yourself off to criticism prevents you from seeing real problems → stay open without agreeing to everything
- **Polite feedback as truth**: Friends/family default to politeness → explicitly ask for honest negative feedback; create questionnaires for external testers
- **Trusting contradictory feedback equally**: When one group loves a mechanic and another hates it, you have less actionable data than consistent feedback → prioritise patterns over outliers
- **Changing mechanics instead of clarifying**: Every unclear mechanic gets mechanically rewritten → often the real issue is explanation, not design
- **Designer participation in blind tests**: Your presence influences player behavior → step back for blind testing to see actual comprehension
- **Ignoring cheating as signal**: Accidental cheating reveals rule ambiguity; intentional cheating by children reveals unintuitive design → pay attention to both

## Application

**When prototyping**: Use card sleeves + printed cards + generic tokens + basic dice. This gets a playable prototype ready in 1-2 days without investing in production.

**During first playtest with friends**: Expect mechanical breaks. Improvise past problems (that's the point). Focus on whether the core experience is fun and if rules make sense.

**During local playtesting**: Observe carefully. Take dated notes on what frustrated players, what confused them, who disengaged. Ask the four feedback questions. Create a questionnaire for external testers.

**During external testing**: Provide minimal designer introduction. Watch how fresh players interpret rules without your guidance. Track which victory paths win, which powers feel overpowered. Ask specifically about rule clarity and mechanical balance.

**When analyzing feedback**: Separate the signal (this is confusing) from the noise (I don't like this theme). Ask clarifying questions to identify root problems. Distinguish between "I don't understand this rule" and "This rule feels unfair."

**When iterating**: Don't be afraid to change drastically. 10+ iterations is normal. The Mandalorian Adventures example shows a core concept (numbered cards on action piles) evolved through 6+ prototypes. Crisis mechanic emerged around prototype 6 as playtesting revealed what the game needed.

**Tradeoff**: Designer participation in local testing (you feel problems firsthand) vs. stepping back in blind tests (fresh perspective without your influence). Both are valid; choose based on what you need to learn.
