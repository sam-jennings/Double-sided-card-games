# Game Design Execution System  
  
> A practical operating system for taking a board game from idea to pitch-ready project.  
>  
> This document is not a replacement for the lesson notes. It is the execution layer that sits on top of them.  
>  
> Use it to decide what to do next, what not to do yet, how to run playtests, when to change direction, and when to move to the next stage.  
  
---  
  
# 1\. Purpose of This Document  
  
The course material gives you:  
- concepts  
- principles  
- case studies  
- cautions  
- business and production realities  
  
This document adds what the course material does not strongly enforce:  
- a step-by-step working process  
- stage gates  
- decision rules  
- iteration discipline  
- a way to translate knowledge into action  
  
The goal is simple:  
  
**Reduce drift. Reduce overthinking. Increase useful iterations.**  
  
---  
  
# 2\. Core Rule of Game Development  
  
Board game development is not a straight line.  
  
It is a loop:  
  
```text  
Define → Build → Test → Diagnose → Decide → Repeat
```

Everything in this document exists to keep you inside that loop.

When you are stuck, ask:

1.  What am I trying to learn right now?
    
2.  What is the fastest way to test that?
    
3.  What single decision should this test help me make?
    

If you cannot answer those questions, stop expanding the game and return to the loop.

# 3\. The Master Workflow

## Stage 0 — Orientation

You do not start by making content.  
You start by defining intention.

Outputs:

-   target player
    
-   core experience
    
-   one-sentence concept
    
-   initial vision document
    

## Stage 1 — Core Concept

You are proving the game has a real center.

Outputs:

-   1-sentence pitch
    
-   3-5 aspirational moments
    
-   first core loop sketch
    

## Stage 2 — Fast Prototype

You are proving the game can exist in playable form.

Outputs:

-   ugly, minimal prototype
    
-   enough components to run a session
    
-   start-to-finish rules skeleton
    

## Stage 3 — Core Loop Validation

You are proving that the central player actions create the intended experience.

Outputs:

-   3-10 meaningful playtests
    
-   notes on confusion, pacing, engagement, and tension
    
-   clear diagnosis of what is and is not working
    

## Stage 4 — Structural Iteration

You are making the game coherent.

Outputs:

-   fewer unnecessary mechanics
    
-   clearer turn flow
    
-   more reliable emotional payoffs
    
-   improved audience fit
    

## Stage 5 — Content Expansion

You are adding variety only after the core works.

Outputs:

-   more cards / roles / scenarios / powers / content modules
    
-   early balancing data
    
-   clearer scope and component list
    

## Stage 6 — Usability and Communication

You are making the game teachable and readable.

Outputs:

-   working rulebook draft
    
-   clear visual hierarchy
    
-   improved iconography / card layout / ergonomics
    

## Stage 7 — Validation and Positioning

You are testing whether the game stands on its own and belongs in a market.

Outputs:

-   blind playtesting
    
-   market comparables
    
-   production assumptions
    
-   pitch materials
    

## Stage 8 — Pitch / Publish Preparation

You are preparing either for publishers or for self-publishing.

Outputs:

-   pitch sheet
    
-   mood / art direction
    
-   mock budget
    
-   next-step plan

# 4\. The Only Daily/Weekly Operating Loop

Use this cycle every week while actively designing a game.

## Weekly Cycle

### Step 1 — Define the current problem

Write one sentence:

-   What is the main thing currently stopping the game from being good?
    

Examples:

-   Players do not understand what decisions matter.
    
-   The traitor has too little room to bluff.
    
-   Downtime is too high at 4 players.
    
-   The game ends before it reaches a climax.
    
-   The theme is decorative, not integrated.
    

### Step 2 — Form one hypothesis

Write one sentence:

-   What specific change am I testing next?
    

Examples:

-   Reducing hand size from 6 to 4 will increase tension.
    
-   Giving the traitor one hidden sabotage action per round will improve ambiguity.
    
-   Interlaced turns will reduce downtime at 4 players.
    
-   Extending the objective track by 2 rounds will allow the game to peak properly.
    

### Step 3 — Build only what is required

Do not redesign everything.  
Build only the smallest version needed to test the hypothesis.

### Step 4 — Run the test

Run 1-3 sessions with the prototype.

### Step 5 — Diagnose what happened

Write:

-   Observed
    
-   Expected
    
-   Mismatch
    
-   Likely cause
    
-   Decision
    

### Step 6 — Make one decision

Choose one:

-   keep
    
-   remove
    
-   simplify
    
-   clarify
    
-   expand
    
-   retheme
    
-   test again
    

Then repeat.

# 6\. Stage Gates

Do not move to the next stage because you feel like it.  
Move only when you have earned it.

## Gate A — Concept to Prototype

Move on only if:

-   you can explain the game in one sentence
    
-   you know who the game is for
    
-   you can describe what players do on a normal turn
    
-   you can name 3-5 aspirational moments
    
-   you know the intended game type: competitive, co-op, team, deduction, etc.
    

If you fail this gate:

-   stop inventing content
    
-   rewrite the concept until it is clear
    

## Gate B — Prototype to Playable

Move on only if:

-   the game can run from start to finish
    
-   the win condition exists
    
-   turns can happen repeatedly without inventing new rules every minute
    
-   components are ugly but functional
    
-   the prototype is fast enough to test
    

If you fail this gate:

-   cut systems
    
-   reduce scope
    
-   focus on the smallest functioning loop
    

## Gate C — Playable to Core Loop Working

Move on only if:

-   players understand what they are trying to do
    
-   there is at least one meaningful choice per turn
    
-   the central mechanic actually creates tension, delight, or interest
    
-   players are not confused about the basic structure
    
-   the game feels like the thing it claims to be
    

If you fail this gate:

-   do not add content
    
-   do not start visual polish
    
-   do not “fix” it with theme text alone
    

## Gate D — Core Loop Working to Expansion

Move on only if:

-   2-3 tests in a row confirm the loop basically works
    
-   you are no longer rewriting core rules every session
    
-   players engage with the intended decisions
    
-   the game reaches some form of payoff or climax
    
-   you know what the game’s strongest element is
    

If you fail this gate:

-   simplify the loop
    
-   remove dead mechanics
    
-   sharpen the decision space
    

## Gate E — Expansion to Polish

Move on only if:

-   the game structure is stable
    
-   content additions are not exposing fundamental design cracks
    
-   the audience fit is clear
    
-   the game length is roughly correct
    
-   the production direction is plausible
    

If you fail this gate:

-   stop adding more cards / factions / scenarios
    
-   fix structure first
    

## Gate F — Polish to Pitch / Publish Prep

Move on only if:

-   the game survives blind or semi-blind testing
    
-   the rulebook can teach the game
    
-   the component list is mostly realistic
    
-   the experience is consistent enough to describe cleanly
    
-   you can position it in the market
    

If you fail this gate:

-   you are not ready to pitch
    
-   you are not ready to self-publish
    
-   return to testing or usability work

# 7\. Execution Checklists by Stage

## Stage 0 — Orientation Checklist

-    Define target player type
    
-    Define primary player motivation
    
-    Define the main emotional outcome
    
-    Decide whether the game is co-op, competitive, team-based, hidden-role, etc.
    
-    Write one-sentence concept
    
-    Write 3-5 aspirational moments
    
-    Decide whether the design is top-down or bottom-up
    

## Stage 1 — Core Concept Checklist

-    Identify the core player loop
    
-    Define the primary action type
    
-    Define how the game ends
    
-    Define what success/failure looks like in one round
    
-    Identify one thing that makes the game worth existing
    

## Stage 2 — Fast Prototype Checklist

-    Use placeholder materials
    
-    Build only what is needed for 1 test
    
-    Make every rule visible somewhere
    
-    Ensure turn order exists
    
-    Ensure scoring / win state exists
    
-    Ensure it can run in under testable conditions
    

## Stage 3 — Core Loop Testing Checklist

-    Run solo dry run
    
-    Run first external playtest
    
-    Track confusion points
    
-    Track engagement peaks and troughs
    
-    Track whether players understand the intended goal
    
-    Record what people talk about after the game
    

## Stage 4 — Structural Iteration Checklist

-    Remove at least one unnecessary thing before adding anything new
    
-    Identify what players enjoy most
    
-    Rebuild around that strength
    
-    Check whether turn flow feels natural
    
-    Check whether the game reaches a climax
    
-    Check whether the audience fit is improving or degrading
    

## Stage 5 — Content Expansion Checklist

-    Add content only after core loop is proven
    
-    Add one content category at a time
    
-    Check whether content adds variety or just volume
    
-    Track whether added content creates balance problems
    
-    Track whether added content makes teaching harder
    

## Stage 6 — Usability and Rules Checklist

-    Write rulebook outline
    
-    Put game flow before edge cases
    
-    Improve icon clarity
    
-    Improve component ergonomics
    
-    Check readability and visual hierarchy
    
-    Check whether the rulebook teaches or only documents
    

## Stage 7 — Validation Checklist

-    Run blind or semi-blind tests
    
-    Track rule misunderstandings
    
-    Track strategic dominance issues
    
-    Check whether players want a rematch
    
-    Compare against market competitors
    
-    Estimate viable MSRP range
    

## Stage 8 — Pitch / Publish Checklist

-    Write pitch sheet
    
-    Write target publisher list or self-publishing plan
    
-    Build realistic component list
    
-    Create mood board / theme & look summary
    
-    Build mock budget
    
-    Write next-step plan with dates

# 8\. The Decision Engine

Every design decision must pass this filter.

## Step 1 — Name the problem

What exact problem are you solving?

Bad:

-   “I want to add more interesting cards.”
    

Good:

-   “Players have too few surprising turns after round 3.”
    

## Step 2 — Name the desired player behaviour

What do you want players to do more often?

Examples:

-   accuse with imperfect information
    
-   take risks
    
-   coordinate under uncertainty
    
-   pivot strategy midgame
    
-   engage during other players’ turns
    

## Step 3 — Name the desired emotional effect

What should it feel like?

Examples:

-   tense
    
-   triumphant
    
-   suspicious
    
-   clever
    
-   desperate
    
-   playful
    

## Step 4 — Predict the consequence

What will this change likely affect besides the target problem?

Examples:

-   longer turns
    
-   more randomness
    
-   less clarity
    
-   more downtime
    
-   better theme integration
    
-   harder teaching burden
    

## Step 5 — Decide whether to test

Only test if:

-   the problem is real
    
-   the change is targeted
    
-   the effect can be observed
    
-   the test is cheap enough
    

If not, do not implement it.

# 9\. The Anti-Drift Rules

These are hard rules.

## Rule 1 — Do not add content to fix boredom

If the game is boring, the core loop is weak.  
More cards will not solve that.

## Rule 2 — Do not polish what you still redesign

If you are still rewriting core mechanics, do not invest in:

-   final layouts
    
-   final illustrations
    
-   premium components
    
-   production detail
    

## Rule 3 — Do not solve balance before fun

An unfun balanced game is still bad.

## Rule 4 — Do not solve theme with flavor text alone

If the mechanic fights the theme, theme loses.

## Rule 5 — Do not interpret confusion as player failure

Repeated confusion is almost always design, communication, or hierarchy failure.

## Rule 6 — Do not change three things at once

You will not know what caused the result.

## Rule 7 — Do not chase trends unless you can move very fast

Follow passion and a clear player promise instead.

## Rule 8 — Do not build for everyone

Choose a primary audience.

## Rule 9 — Do not use “it will work later” as a defense

If it does not work now, it is not working.

## Rule 10 — Do not confuse effort with progress

Spending time is not the same as learning something.

# 10\. The Core Questions You Must Revisit Constantly

## Player Questions

-   Who is this game for?
    
-   What kind of player would dislike this?
    
-   What motivation am I optimizing for: winning, fun, socializing, storytelling?
    
-   What will players remember after the game?
    

## Experience Questions

-   What should players feel in the first 5 minutes?
    
-   What should players feel in the final 10 minutes?
    
-   What are the 3 strongest moments this game produces?
    
-   Are those moments frequent enough?
    

## Mechanics Questions

-   What decisions actually matter?
    
-   Which rules are doing the most work?
    
-   Which rules are adding weight without payoff?
    
-   Is complexity creating depth, or only friction?
    

## Theme Questions

-   Do player actions make sense in the game world?
    
-   Does the win condition feel thematically right?
    
-   Is the theme actually integrated, or pasted on?
    

## Usability Questions

-   Can players see what matters?
    
-   Are icons reinforcing text, or replacing clarity?
    
-   Does the turn structure feel natural?
    
-   Could a new player learn this from the rulebook?
    

## Business Questions

-   Is this component list viable at my target MSRP?
    
-   Does the game’s scope fit its likely audience and market?
    
-   Would a publisher in this space plausibly want it?
    
-   If self-publishing, can I realistically execute the rest of the business?

# 11\. Mapping Mechanics to Experience

Use this template whenever you add or evaluate a mechanic.

```
Markdown

Mechanic:  
What players do:  
What decision it creates:  
What emotion it should create:  
What risk it introduces:  
Why the game needs it:  
What happens if removed:
```

## Example

```
Markdown

Mechanic: Hidden sabotage card  
What players do: One player secretly chooses whether to help or hinder  
What decision it creates: Bluff now or stay concealed  
What emotion it should create: Suspicion and second-guessing  
What risk it introduces: Can feel arbitrary if players have no deduction tools  
Why the game needs it: Core identity is traitor tension  
What happens if removed: Game becomes open optimization puzzle
```

If you cannot fill this out cleanly, the mechanic is not ready.

# 12\. Playtesting Protocol

## Stage 1 — Solo Dry Run

Purpose:

-   verify the game runs
    
-   catch obvious breakdowns
    
-   estimate session length
    

Do not judge the design too hard yet.  
You are only checking whether it can physically happen.

## Stage 2 — First Real Test

Purpose:

-   see whether the game feels like the intended concept
    
-   identify the first major confusion or failure points
    

Rules:

-   expect things to break
    
-   improvise through dead ends
    
-   record where the game stops making sense
    

## Stage 3 — Local Iteration Testing

Purpose:

-   improve structure
    
-   refine pacing
    
-   diagnose audience fit
    

Record:

-   where players light up
    
-   where they disengage
    
-   what they misunderstand
    
-   whether they want to play again
    

## Stage 4 — External / Blind Testing

Purpose:

-   prove the game stands without you
    
-   validate clarity, balance, and teaching quality
    

Do not help too quickly.  
Watch what the design and rules fail to communicate.

# 14\. How to Read Feedback Correctly

## Treat feedback as data, not orders

Players are usually best at describing:

-   what felt bad
    
-   what confused them
    
-   what excited them
    

They are usually worse at designing the fix.

So separate:

-   **problem reports**  
    from
    
-   **proposed solutions**
    

## The feedback triage model

### Keep

When:

-   multiple players independently report the same issue
    
-   observed behavior matches the complaint
    
-   it points to a core experience problem
    

### Clarify

When:

-   issue seems to be understanding, not design
    
-   players only struggle because wording / hierarchy / explanation is weak
    

### Ignore for now

When:

-   feedback is contradictory
    
-   player is reacting to something outside target audience
    
-   change would force major scope shift without strong evidence
    
-   the issue is about unfinished polish you are not testing yet

# 15\. The Clarity vs Design Test

When players dislike something, ask:

## Is it a clarity problem?

Examples:

-   they forgot a step
    
-   they misunderstood icon meaning
    
-   they thought a rule worked differently
    
-   they did not realize a choice existed
    

Response:

-   improve rulebook
    
-   improve card text
    
-   improve layout
    
-   improve turn summary
    
-   improve examples
    

## Is it a design problem?

Examples:

-   they understood the rule but still found it boring
    
-   the optimal choice is obvious every time
    
-   the mechanic drags pace down
    
-   the risk/reward balance feels bad
    
-   the game ends before payoff
    

Response:

-   redesign the mechanic
    
-   simplify the system
    
-   restructure the loop
    
-   retime the payoff
    

Do not use mechanical changes to fix what clear communication would solve.

# 16\. How to Avoid Complexity Bloat

Complexity comes from:

-   number of subsystems
    
-   weight of each subsystem
    
-   fiddliness / tracking burden
    

Whenever the game starts feeling heavy, run this audit:

## Subsystem audit

List every subsystem.  
Ask:

-   Is this essential?
    
-   Does this create a distinctive decision?
    
-   Could another system already cover its job?
    

## Rule weight audit

For each subsystem, ask:

-   How hard is this to teach?
    
-   How often does it matter?
    
-   Is the payoff worth the explanation cost?
    

## Fiddliness audit

Ask:

-   How many component types are being tracked?
    
-   How often do players need to move, rotate, count, or remember things?
    
-   Could physical handling be simplified?
    

## The kill question

If I removed this rule, would the game lose identity or just lose weight?

Cut aggressively.

# 17\. Interaction and Scaling Rules

## Multiplayer

Downtime is one of the biggest threats to enjoyment.

Use:

-   interlaced turns
    
-   simultaneous choices
    
-   reactions during others’ turns
    
-   lower volatility between turns
    

Watch for:

-   phone checking
    
-   disengagement
    
-   no one caring during other turns
    

## Direct interaction

Use when:

-   drama, conflict, politics, or accusation are core to the game
    

Avoid or limit when:

-   target audience dislikes confrontation
    
-   game becomes kingmaking-heavy
    
-   a single targeted effect ruins the experience
    

## Cooperative

Good co-op usually needs:

-   more to do than time allows
    
-   interdependence
    
-   individual importance
    
-   team strategy with individual agency
    
-   quarterback prevention
    

## Solo

Works best when:

-   the game is already co-op
    
-   there is a strong automated opponent
    
-   the player still faces meaningful uncertainty

# 18\. Theme Integration Rules

Theme should do more than decorate.

## Audit for thematic consistency

Ask:

-   Who are the players in the fiction?
    
-   What are they trying to achieve?
    
-   Do their actions make sense?
    
-   Does the game structure match the theme?
    
-   Does the victory condition feel narratively correct?
    

## Choose narrative mode

Use **scripted** when:

-   specific reveals matter
    
-   sequence matters
    
-   strong authored story is the point
    

Use **emergent** when:

-   replayability matters
    
-   player-authored story is the point
    
-   system interactions create the narrative
    

Use **hybrid** when:

-   you want structure without losing variety
    

## Emotional design rule

Pick the emotion first, then align:

-   writing
    
-   mechanics
    
-   pacing
    
-   component feel
    
-   visual tone

# 19\. Usability and Communication Rules

## Rulebook sequence

1.  Who are the players?
    
2.  What are they trying to do?
    
3.  How does the game broadly work?
    
4.  What components exist?
    
5.  How does setup work?
    
6.  What happens in a round / turn?
    
7.  What are the detailed rules?
    
8.  How does the game end?
    
9.  What edge cases need clarification?
    

## Rulebook standards

-   short sentences
    
-   minimal jargon
    
-   game flow before edge cases
    
-   visuals for setup
    
-   examples where ambiguity is likely
    
-   index or reference support for heavier games
    

## Card and component usability

-   top-left is premium information space
    
-   do not rely on color alone
    
-   redesign the card before shrinking the font too far
    
-   use icons to reinforce, not replace, clarity
    
-   prioritize readability over atmosphere

# 20\. Visual and Production Discipline

## During development

The prototype only needs enough visual structure to test:

-   hierarchy
    
-   readability
    
-   action comprehension
    

It does not need:

-   final art
    
-   premium materials
    
-   perfect polish
    

## When thinking about production early

Ask:

-   Can this hit a plausible MSRP?
    
-   Am I accidentally designing something too expensive?
    
-   Are there custom components I do not actually need?
    
-   Would standard sizes and stock components solve this cheaper?
    

## The backward viability check

Target MSRP ÷ 5 = approximate target production cost

If your scope cannot fit that:

-   reduce components
    
-   change materials
    
-   increase price expectation
    
-   target a different publishing route

# 21\. Publisher vs Self-Publishing Decision

## Pitch to publishers if:

-   you want to focus on design
    
-   you do not want to run a business
    
-   you are comfortable giving up some control
    
-   the game fits a known catalog niche
    

## Self-publish if:

-   you want full control
    
-   you are willing to manage production, freight, customer service, marketing, and cash flow
    
-   the game needs a specific visual or product identity
    
-   you are building a company, not just making one game
    

## Warning

Do not romanticize self-publishing.  
It is not “publishing but better.”  
It is building and running a business.

# 22\. Market Positioning Rules

Before pitch stage, write:

## Comparable games

List 3-5 games that are:

-   mechanically relevant
    
-   audience relevant
    
-   commercially relevant
    

For each, answer:

-   Why does it succeed?
    
-   Where does your game overlap?
    
-   Where does your game differ?
    
-   What gap does your game fill?
    

## Unique selling proposition

This is not “my game is unique.”  
It is:

-   why someone would choose it
    
-   why a publisher would care
    
-   why the market has room for it
    

Good USP examples:

-   familiar structure with a new emotional experience
    
-   shorter version of a longer genre
    
-   more accessible version of a beloved style
    
-   a stronger thematic integration than comparable games

# 23\. Pitch Readiness Test

You are ready to pitch only if you can answer clearly:

1.  Who are you in the game?
    
2.  What do you do?
    
3.  How do you win?
    
4.  Who is it for?
    
5.  Why does it deserve to exist?
    
6.  What does it cost to make, approximately?
    
7.  Which publishers or channels make sense for it?
    

## Pitch health check

-   Can someone retell your game after hearing it once?
    
-   Does the description lead with experience, not only mechanics?
    
-   Is the component list specific enough to sound real?
    
-   Are your numbers estimates, not fake certainty?
    
-   Does your passion sound genuine?

# 24\. The “What Should I Do Next?” Resolver

Use this whenever you are lost.

## If you do not know what the game is yet:

Return to:

-   target player
    
-   core experience
    
-   aspirational moments
    
-   one-sentence concept
    

## If the game exists but is boring:

Return to:

-   core loop
    
-   meaningful choices
    
-   tension and payoff
    
-   interaction and pacing
    

## If the game is interesting but messy:

Return to:

-   subsystem audit
    
-   turn-flow simplification
    
-   clarity vs design diagnosis
    

## If the game works but feels flat:

Return to:

-   emotional design
    
-   memorable moments
    
-   spotlight moments
    
-   stronger player identity
    

## If the game is fun but confusing:

Return to:

-   rulebook structure
    
-   card layout
    
-   icons
    
-   examples
    
-   player aids
    

## If the game seems ready but you feel uncertain:

Return to:

-   blind testing
    
-   market comparables
    
-   pitch sheet clarity
    
-   component cost sanity check

# 25\. The Weekly Review

Once per week, answer these:

## Project Review

-   What did I learn this week?
    
-   What is the main unresolved problem?
    
-   What stage is the game in now?
    
-   What am I tempted to do too early?
    
-   What evidence says the game is improving?
    
-   What evidence says I may be fooling myself?
    

## Decision Review

-   What did I change?
    
-   Was that change based on evidence or taste?
    
-   What is the next hypothesis?
    
-   What am I deliberately postponing?
    

## Discipline Review

-   Am I still in the design loop?
    
-   Or am I polishing, fantasizing, or expanding to avoid the hard problem?

# 26\. The Personal Rules of Good Development

These are the habits this system is trying to create.

-   Build fast.
    
-   Test ugly.
    
-   Observe honestly.
    
-   Change one thing at a time.
    
-   Cut more than you add.
    
-   Do not confuse explanation with experience.
    
-   Do not confuse theme with integration.
    
-   Do not confuse content with design.
    
-   Do not confuse enthusiasm with clarity.
    
-   Do not move on because you are bored.
    
-   Move on because the design has earned it.

# 27\. One-Page Quickstart Version

Use this if starting a new game today.

## Step 1

Write:

-   target player
    
-   core emotion
    
-   one-sentence concept
    
-   3 aspirational moments
    

## Step 2

Choose:

-   top-down or bottom-up
    
-   co-op / competitive / team / hidden-role / etc.
    

## Step 3

Define:

-   what players do on a normal turn
    
-   how they win
    
-   how the game ends
    

## Step 4

Prototype:

-   enough to run one full session
    
-   nothing more
    

## Step 5

Run:

-   solo test
    
-   one external test
    

## Step 6

Record:

-   what worked
    
-   what confused
    
-   what was exciting
    
-   what was dead
    

## Step 7

Choose one change only.

## Step 8

Repeat until:

-   players understand the loop
    
-   the loop produces the intended feeling
    
-   players want another game
    

Only then:

-   add content
    
-   refine visuals
    
-   write better rules
    
-   think seriously about pitch or production
